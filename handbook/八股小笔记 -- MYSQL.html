    八股小笔记 -- MYSQL

    # 八股小笔记 -- MYSQL

    # B树和B+树的区别，Mysql为什么使用B+树

    # B+树特点

    1. 非叶子节点子树指针与关键字个数相同
    2. 所有叶子节点增加一个指针
    3. 叶子节点包含所有关键字
    # B+树优势
    • 磁盘读写代价更低
      • B+树内部节点没有指向关键词的指针，内部节点更小
      • 一次性读入内存需要查找的关键字越多，读写次数越小
    • 查询效率更稳定
      • 任何关键字的查找都是从根节点到子节点，由于所有叶子节点在同一层，so查询路径长度相同，使每个数据查询效率相当
    • 便于范围查询
      • 优化B树元素遍历效率低下的问题，其只需遍历叶子节点就可实现整棵树的遍历
    # Mysql使用B+树
    • Mysql中MyIsAM和InnoDB采用B+树结构，前者是非聚集索引，后者是聚集索引
    • B+树节点只存储索引Key值，具体信息地址存于叶子节点中
    • Mysql中的InnoDB页是一个B+树节点，一个InnoDB页默认16kb，一般一颗两层的B+树可以存2000万行左右数据，可以很好支持全表扫描，范围查找等SQL语句
    # ISAM
    • 索引顺序存取方法，可以按照进入顺序或根据索引访问，每个索引定义一次不同排列的记录
    # MyISAM
    • Mysql5.1之前的默认存储引擎，基于ISAM
    • 每个MyISAM在磁盘存储成3个文件，都以表名开始
    • 有三种文件类型
      • .frm -- 表定义
      • .MYD -- 表数据
      • .MYI -- 表索引
    # InnoDB
    • Mysql默认存储引擎，支持ACID兼容的事务功能，类似于PostgreSQL
    • 有.frm存放元数据信息和表结构的定义信息
    • .ibd或.ibdata存放InnoDB数据，区别为是否共享表空间
      • 独享用.ibd，每个表一个
      • 共享用.ibdata，所有表共用一个
    • InnoDB事务采用两阶段提交模式:prepare,commit
    # MyISAM于InnoDB差别
    • InnoDB支持事务，支持外键，MyISAM都不支持
    • InnoDB主键使用聚簇索引，辅助索引（在聚簇索引之上创建的索引）使用非聚簇索引，MyISAM都是非聚簇索引
    • InnoDB不保存表具体行数，查找数据需全表遍历
    • InnoDB最小锁粒度为行锁，比表锁粒度更小，避免查询和更新时被阻塞，导致并发访问受限
    • 大数据量排序、全表扫描、count等操作，MyISAM更优

    # InnoDB是怎么实现事务的

    • 概述：InnoDB概念

    • 通过Buffer Pool, LogBuffer, Redo Log, Undo Log来实现事务
      • 执行语句前，先缓存到Buffer Pool
      • 之后修改Buffer Pool中数据
      • 生成RedoLog对象，存入LogBuffer中
      • 生成undoLog日志
      • 若事务提交，则将RedoLog对象持久化，后利用其他机制持久化到磁盘中
      • 若事务回滚，利用undoLog回滚
    # 持久化
    • 把域对象永久保存到数据库中
    • 持久化封装了数据访问细节，提供api接口
    • 利用持久化可以减少访问数据库次数，增加app运行速度
    • 重用性高
    • 耦合度低，不依赖底层数据库与上层业务逻辑实现，更换数据库不用修改代码

    # MYSQL索引

    通过索引对数据进行排序，降低数据排序的成本，（间接）降低CPU损耗

    # Mysql索引底层数据结构

    B+树

    • 叶子节点存具体数据，非叶子节点存储指针（导航）
    • 一个节点可有多个叶子节点，整体结构矮胖
    • 叶子节点互相有双向链表，便于扫库
    # Mysql聚簇索引与非聚簇索引
    • 聚簇索引：将数据与索引放在一起，必须有且只能有一个
      • 若存在主键，则该主键就是聚集索引
      • 若不存在主键，将UNIQUE唯一索引作为聚集索引
      • 都不存在，InnoDB自动生成rowId作为隐藏聚集索引
    • 非聚簇索引（二级索引）：数据索引分开，叶子节点无数据，存储数据地址，可存在多个

    # 回表

    使用非聚簇索引，为取到具体数据，通过PK主键回到聚集索引去查询数据

    # 覆盖索引

    通过索引查询的数据，包含了所有要查询的数据，无需回表

    # 索引失效

    1. 违反最左前缀法则：跳过最左字段匹配
    2. 范围查询右边的列
    3. 在索引列上进行运算操作（函数）
    4. 字符串不加''（类型转换）
    5. 头部模糊查询/头尾模糊查询

    # 索引基本原理

    • 避免整张表遍历，将无序数据转为有序
    • 对排序结果生成倒排表
    • 倒排表内容拼接数据地址链
    • 查询时，先拿倒排表内容，再取数据地址，从而拿到数据
    # 倒排表
    • 有递增id的hash表，key为单词，value为该词数据的id

    # SQL优化

    # 表设计优化

    选择合适的数据类型

    # SQL语句优化

    1. 避免使用SELECT *
    2. 避免造成索引失效
    3. 用UNION ALL代替UNION，避免UNION去重的多一次过滤
    4. 避免where语句中出现表达式操作（避免索引失效）
    5. 能用INNER JOIN 就不要用LEFT JOIN，（一定要用LEFT JOIN的情况，主表尽量为小表）

    # 主从复制/读写分离

    # MYSQL主从集群

    • 搭建MYSQL主从集群，主库实现写操作后记录到Binlog中，从库提取Binlog文件写入到从库的Relaylog中，实现主从数据同步
    • 只能由主节点向从节点进行苏数据同步，过程异步，有数据丢失风险
    • 可在传输过程中加入响应机制实现半同步复制，牺牲了一点点同步速度，保证数据安全性
    # 并行复制 MTS
    • MYSQL5.6之后为解决主从复制延迟而设计的功能
    • 将从库两个线程IO Thread和SQL Thread的工作模式改为多线程机制
      • MYSQL5.6：基于库
      • MYSQL5.7：基于组（真正的并行复制），所有处于prepare阶段的事务互相不冲突，可并行提交,引入新变量slave-parallel-type,可以配置并行方式
      • MYSQL8.0：基于write-set，额外由一个集合变量存储事务修改的记录信息，提交时将记录打包成事务，将事务写入磁盘，提交事务将修改的主键值与集合变量对比，判断是否冲突，无冲突即并行，粒度达到row级别

    • 顾名思义，查询SQL太慢
    • 危害：
      • 用户体验差、占内存影响性能、DDL操作阻塞

      • long_query_time:定义慢查询时间（一般为2s）

    EXPLAIN/DESC命令：查看SELECT语句查询计划 （key：命中索引 ； key_len：索引占用大小；type:SQL连接类型；extra:是否出现回表情况）

    1. 聚合查询
    2. 多表查询
    3. 表数据量过大查询
    4. 深度分页查询
    # explain语句
    key 含义
    id select的序列号，值越大优先值越高，某些子查询会被优化为join，id值相同
    select_type 对应查询类型
    1.simple（简单select，不用union和子查询）
    2.primary（包含复杂字部分）
    3.union（union后的第二个和随后的select）
    4.union result（union结果）
    5.dependent union（外部查询之后，union后的第二个select）
    6.subquery（子查询第一个select）
    7.dependent subquery（外部查询之后，子查询第一个select）
    8.deriver（派生表的select，FROM子句的子查询）
    9.unCacheable subquery（表示查询结果不能被缓存）
    table 表名
    type 查询方式
    possible_keys 可能用到的索引
    Key 实际索引
    key_len 实际索引长度
    ref 连接匹配条件
    rows 预估记录条数
    Extra 额外信息
    # OPTIMIZE TABLE整理碎片
    • 使用OPTIMIZE语句，MYSQL会锁定表
    • 主要步骤：
      1. 检查，删除空闲空间与索引
      2. 重新组织表数据，使紧凑
      3. 优化表索引
      4. 更新表信息
    • 用于消灭聚簇索引所带来的内存碎片

    # 事务

    # 事务的基本特性 ACID

    • Atomic（原子性）：包含整个数据库，任意错都回滚
    • Consistency（一致性）：不能破坏关系数据完整性及业务逻辑一致性
    • Isolation（隔离性）：并发时，存在事务隔离
    • Durability（持久性）：事务成功结束，数据库更新需永久保存，且数据能恢复

    # 脏读，不可重复读，幻读

    事务并发导致的“小”问题，可以加锁，事务隔离，MVCC来处理

    • 脏读：读取未提交的数据。
    • 不可重复读：同一个事务中多次执行同一个select, 读取到的数据发生了改变（改数据）
    • 幻读：同一个事务中多次执行同一个select, 读取到的数据行发生改变（增删数据）

    # 事务隔离级别

    • 读未提交READ UNCOMMITTED (RU)：幻读，不可重复读和脏读均允许；
    • 读已提交 READ COMMITTED (RC)：允许幻读和不可重复读，但不允许脏读
    • 可重复读 REPEATABLE READ (RR)：允许幻读，但不允许不可重复读和脏读 （ MYSQL默认隔离级别 ）
    • 串行 SERIALIZABLE: 幻读，不可重复读和脏读都不允许

    # undo log/redo log

    缓冲池buffer pool：将经常使用的数据，缓存到缓冲池

    数据页page：InnoDB存储引擎磁盘管理的最小单元，每页默认16KB，页存储行数据

    • redo log
      • 重做日志，记录数据页物理修改，实现事务持久性
      • 在将日志刷新到磁盘的过程中有误，则需redo log日志进行恢复
    • undo log
      • 回滚日志，逻辑日志，记录相反的指令，用于回滚，实现原子性和一致性

    # mysql锁

    • 按照锁粒度来划分可分为：行锁、表锁、全局锁
    • 锁的作用顾名思义，行锁目前InnoDB支持，表锁MyISAM和InnoDB都支持
    • 每种锁都含有共享锁和排他锁
      • 共享锁（S）： 多事务可对同一个数据共享同一把锁，只读不能改
      • 排他锁（X）：一事务一把锁，InnoDB会对增删改自动添加排他锁
    • 表锁还额外加了一种意向锁，InnoDB意向锁
    • 全局锁：Flush tables with read lock。加锁后整个数据库都只读，所有数据变更操作被挂起，一般用于全库备份

    # InnoDB意向锁

    • 为支持多粒度锁而设计的表级锁，不与行锁和同级意向锁冲突
    • 共分为意向共享锁（IS）与意向排他锁（IX）
    • 为防止事务获取表数据时一行行判定是否锁定而产生的效率低下，实现了对应表的意向表锁，若锁被其他事务拿取，则可以直接去排队申请资源
    • 冲突列表
    意向共享锁(IS) 意向排他锁(IX) 表级共享锁(S) 表级排他锁(X)
    意向共享锁(IS) 兼容 兼容 兼容 不兼容
    意向排他锁(IX) 兼容 兼容 不兼容 不兼容
    表级共享锁(S) 兼容 不兼容 兼容 不兼容
    表级排他锁(x) 不兼容 不兼容 不兼容 不兼容

    # MYSQL 的MVCC

    • 名为多版本并发控制，读写隔离，使读写操作没有冲突
    • 修改同一条数据时，会在undo log日志中，记录一条记录版本链表，每条数据含回滚版本号，版本id等
    • readView（当前读）
      • 每次查询加锁，势必拿到最新数据
    • readView（快照读）
      • 在开始事务时创建readView，以数组形式记录活动事务（未提交事务）的事务id
      • 访问数据时，比对readview中的id，若比readview都小，则可以访问，若比readview都大，则不可访问，若在readview中，则获取roll_pointer,取上一版本，再来一次上述操作
      • RC隔离级别下的事务每次查询前都生成新的readview；RR级别下的readview可复用；序列级别：快照读退化为当前读
